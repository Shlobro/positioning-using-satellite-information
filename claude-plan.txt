================================================================================
SYSTEM OVERVIEW
================================================================================

Nadir assumption: the drone camera points directly down. This means the
drone frame is already near-nadir. No perspective correction for oblique
angles is needed. The normalization we need is:
  1. Scale: match the drone frame's ground footprint to the reference tile
  2. Rotation: align north using compass heading

Each drone packet -> server pipeline:
  1. Receive: {frame, altitude_m, heading_deg, prior_lat, prior_lng}
  2. Crop reference tile from local satellite map around prior position
  3. Normalize drone frame: scale to match footprint, rotate to north-align
  4. Run localizer (heatmap model or dense matcher) to find offset inside crop
  5. Convert pixel offset to meters to lat/lng delta
  6. Fuse into particle filter to smooth estimates over time
  7. Return: {lat, lng, confidence_m, match_score}

Phase 2A -- pretrained dense matcher (RoMa, zero-shot):
  - Evaluated before any custom training
  - Fastest honest signal: what can off-the-shelf neural matching do right now?
  - Result determines whether Phase 2B fine-tunes RoMa or trains a scratch model

Phase 2B -- first trained model (decision-driven):
  - If RoMa zero-shot is close to target: fine-tune RoMa with position-only loss
  - If RoMa zero-shot is clearly inadequate: train crop-conditioned heatmap model
  - If both are useful: heatmap as coarse stage, RoMa as fine-alignment refiner
  Decision recorded in change-log.md before any training code is written.

PROVISIONAL TARGET THRESHOLDS (public-dataset phase):
  Phase 1 (classical baseline):  Record Recall@10m and p90 on UAV-VisLoc val.
    Do NOT set a numeric gate until the first run completes.
  Phase 2A (RoMa zero-shot):     Recall@5m improves >= 0.10 over Phase 1 OR
                                   Recall@10m >= 0.50.
    If neither is met: record failure path in change-log.md and proceed to
    Phase 2B scratch model regardless -- zero-shot failure is useful signal.
  Phase 2B (first trained model): Recall@5m > best Phase 2A result + 0.05,
                                   p90 improves >= 10%.
  Phase 3 (fine-tuned on local):  recall_5m > 0.55, p90 < 15 m.
    SUPERSEDED by local-data gate once real flights exist:
    local recall_5m > public-dataset recall_5m - 0.10
    (we expect some drop on real data; more than 0.10 indicates distribution gap)
  Phase 4 (particle filter):      track_median_error_m < single-frame median,
                                   estimate_jitter_m < 2 m while hovering,
                                   ECE < 0.15 (confidence must be calibrated)
  Phase 5 (live flight):          recall_5m > 0.45 on real data
    This is the first gate that is not provisional -- it reflects deployment
    reality, not a public benchmark.

Rule: once local flight data exists, local-data gates replace public-dataset
gates for all decisions. Public-dataset results are benchmarks; local results
are decisions.

0-C. Implement frame_normalizer.py:
      NADIR ASSUMPTION: camera points directly down. No perspective correction.
      The normalization is purely scale and rotation:

        ground_footprint_width_m = 2 * altitude_m * tan(camera_hfov_rad / 2)
        scale_factor = tile_resolution_mpx / (frame_width_px / ground_footprint_width_m)
        # resize drone frame so 1 px = same real-world meters as reference tile
        normalized = cv2.resize(frame, ...)
        # rotate so North is up (satellite tile is North-up by convention)
        normalized = cv2.warpAffine(normalized, rotation_matrix(-heading_deg), ...)

      - Configurable camera_hfov_deg (default 84 deg, DJI-class cameras)
      - Expose camera calibration via config so it can be updated per-drone
      - Unit test: 0 deg and 360 deg heading must give identical output
      - Unit test: at altitude=50 m, FOV=84 deg, check footprint = ~92 m

      GEOMETRY UNCERTAINTY -- the code is short but the inputs are not clean:
        - Altitude: barometric or rangefinder reads AGL above takeoff point,
          not above the current terrain. On sloped or varied ground this can be
          off by several meters. At 15 m altitude, a 2 m terrain error = 13%
          scale error = ~2 m positional bias before any model runs.
        - FOV: nominal spec (e.g. "84 deg") may differ from actual calibrated FOV
          by 2-5 deg, especially after firmware updates. A 3 deg FOV error at
          30 m altitude introduces ~1.5 m scale error.
        - Heading: compass heading is typically +/-2 deg for a consumer gimbal.
          At 50 m footprint width, 2 deg misalignment = ~1.7 m edge misalignment.
        - Gimbal nadir: a "downward-facing" camera may have 1-3 deg of residual
          tilt. This produces a small parallax shift. At 30 m altitude and 2 deg
          tilt, the center shift is ~1 m. This is within GPS label noise so it
          is acceptable to ignore for v1 but should be measured on real footage.
        - Map GSD mismatch: satellite tile GSD varies by zoom level and latitude.
          Always verify tile_resolution_mpx at runtime, do not hardcode it.

      Implement geometry_sensitivity_report() in geo_utils.py:
        Input: nominal altitude_m, heading_deg, camera_hfov_deg
        Output: table of induced errors for +/-5%, +/-10% perturbations on each
        Run this at startup and log the table to the run artifact folder.
        This is not a test that passes or fails -- it is a diagnostic that makes
        the uncertainty visible before any model result is interpreted.

PHASE 4 -- PARTICLE FILTER TEMPORAL FUSION
================================================================================
Goal: fuse per-frame estimates over time. Reduce jitter and reject outliers.

---- AI AGENT TASKS --------------------------------------------------------

4-A. Implement src/tracking/particle_filter.py:
      State vector: (lat, lng) with optional heading extension
      N=500 particles, initialized as uniform disk of 100 m around prior

      Prediction step:
        Propagate by dead-reckoning delta (lat/lng per timestep)
        Gaussian process noise scaled to dead-reckoning drift (default 0.5 m/s)
        Optionally propagate heading separately if compass is available

      Update step:
        Weight particles by Gaussian likelihood of measurement:
          w_i *= exp(-dist(particle_i, estimate)^2 / (2 * sigma^2))
          where sigma = calibrated_confidence_m from the model
        Normalize weights
        Resample when effective N < N/2

      Failure handling: if confidence < min_threshold, skip update (coast)
      Output: weighted mean (lat, lng) + std as confidence_m

      Tracking metrics: compute jitter_m as stddev of estimates over a
      10-frame window, drift_over_10s_m as displacement from ground truth
      averaged across hover sequences.

4-B. Add confidence calibration to eval pipeline:
      - Fit isotonic regression: (model_score, altitude_m) -> true_error_m
      - Use calibrated sigma in particle filter update
      - Compare RUN with raw vs. calibrated confidence

4-C. Implement src/server/localizer.py:
      - Maintains ParticleFilter instance per session_id
      - Per packet: normalize frame -> generate crop -> run model ->
        update filter -> return estimate
      - If confidence < hard_threshold for 3+ consecutive frames:
        set localization_status = "lost", coast on dead-reckoning

4-D. Implement src/server/receiver.py:
      - asyncio WebSocket server
      - Packet format (JSON + base64 JPEG -- dev protocol, not production):
          {session_id, timestamp_ms, frame_b64, altitude_m, heading_deg,
           prior_lat, prior_lng}
      - Response: {lat, lng, confidence_m, match_score, localization_status, fps}
      - Note: base64/JSON is used here for ease of debugging and integration
        testing. For production, replace with binary framing or gRPC.

4-E. Add sequence evaluation to eval.py:
      - Replay a full flight sequence through localizer.py (offline mode)
      - Compute track_median_error_m, jitter_m, relocalization_time_s
      - Plot position trace overlaid on satellite tile

4-F. Run RUN-008: sequence mode with particle filter on held-out flight.
     Compare to frame-by-frame mode from Phase 3.

4-G. Tune particle filter:
      - Sweep N in {100, 200, 500, 1000}, noise_sigma in {0.1, 0.3, 0.5, 1.0}
      - Log each sweep as a separate run in experiment-log.csv (RUN-009 etc.)
