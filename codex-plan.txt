Satellite-Drone Localization Vertical Implementation Plan

Objective
Build a system that localizes a drone in GNSS-denied conditions using only:
- a downward-looking camera frame
- altitude
- heading / compass
- an approximate prior location within a radius of about 100 meters

The operational assumption is that the drone streams packets to a ground computer. The ground computer performs map lookup, frame-to-map matching, temporal filtering, and outputs an estimated global position with uncertainty.

This plan is vertical: every phase ends with something runnable and reviewable. Each phase has:
- AI Agent responsibilities
- Human responsibilities
- runnable output
- success metrics
- exit criteria

The plan is opinionated. It is based on the research in `research/` and is optimized for:
- server-side compute
- small local search window due to the 100 m prior
- low-altitude nadir imagery
- incremental validation over ambitious but untestable architecture

==================================================
1. Technical Direction
==================================================

Core design choice
Do not start with a giant end-to-end network that predicts latitude and longitude directly from raw pixels.

Start with a staged neural localization system:
1. build the ingestion and evaluation scaffold
2. solve local crop selection using height + heading + prior
3. build a classical baseline
4. add a learned matcher / learned localizer
5. add temporal filtering
6. improve robustness with more data and better models

Why
- The 100 m prior removes the hardest global retrieval problem.
- The main challenge is precise local alignment inside a small map crop.
- A vertical build with a strong baseline will show quickly whether the data quality is good enough.
- At low altitude, failure modes are often caused by data mismatch and weak texture, not only model weakness.

Recommended first target architecture
Input packet:
- timestamp
- frame
- altitude_agl_m
- heading_deg
- optional prior_lat, prior_lon from dead reckoning

Server pipeline:
1. ingest packet
2. normalize frame geometry using the simplest justified model
3. derive expected ground footprint from altitude + camera calibration
4. crop satellite / orthophoto search region around prior
5. align frame and crop into a common heading convention
6. estimate pose of frame inside crop
7. produce measurement score and uncertainty
8. fuse over time with particle filter
9. output estimated position and confidence

Initial matching strategy
- Baseline: ORB or AKAZE + RANSAC homography / affine fit
- First neural stage: evaluate a pretrained dense matcher zero-shot on local crops
- Trained neural stage: either fine-tune the matcher or train a crop-conditioned localizer if the zero-shot model is inadequate
- Temporal stage: particle filter using the per-frame score as likelihood

Important constraint from the research
- Matching at 5-15 m altitude over weak-texture terrain may fail regardless of model quality.
- Use high-resolution orthophotos when possible, not only coarse satellite imagery.
- Expect the best progress first in urban, suburban, and agricultural areas with stable structure.

4. Metrics Framework
==================================================

Do not use only one metric. This system needs accuracy, robustness, latency, and uncertainty quality.

4.1 Frame-level localization metrics
- position_error_m
  Euclidean distance between predicted and true location on the map.
- recall_at_1m
- recall_at_3m
- recall_at_5m
- recall_at_10m
- median_error_m
- p90_error_m
- failure_rate
  Fraction of frames with no valid estimate or confidence below threshold.

4.2 Tracking metrics
- track_median_error_m
- track_p90_error_m
- drift_over_10s_m
- relocalization_time_s
  Time needed to recover after deliberate perturbation or lost frames.
- estimate_jitter_m
  Frame-to-frame variance while hovering or moving slowly.

4.3 Runtime metrics
- end_to_end_latency_ms
- frame_preprocess_latency_ms
- matching_latency_ms
- filter_latency_ms
- throughput_fps

4.4 Confidence / calibration metrics
- confidence_vs_error_correlation
- calibration_curve
- expected_calibration_error
- percent_of_large_errors_with_low_confidence

4.5 Dataset coverage metrics
- number_of_flights
- number_of_frames
- number_of_unique_locations
- number_of_altitude_bands
- number_of_area_types
- season coverage
- lighting coverage

4.6 Primary milestone metric by stage
- Early baseline stages: Recall@5m and median_error_m
- Tracking stages: track_median_error_m and relocalization_time_s
- Production readiness stages: Recall@5m, p90_error_m, failure_rate, end_to_end_latency_ms

5. Dataset Strategy
==================================================

You cannot wait for a perfect custom dataset before coding. Use a layered data strategy.

Data sources
1. Public datasets for initial development
   Candidate sources from research:
   - UAV-VisLoc
   - DenseUAV
   - WildNav-related public data if useful

2. Controlled local dataset collected by you
   This becomes the most important dataset because it matches your camera, altitude band, and terrain.

3. Synthetic pair generation
   Use map crops and transformed drone-like views for debugging and smoke tests only, not as the final proof.

Dataset split policy
- Split by flight / location, not random frames only.
- Keep one held-out area for honest generalization testing.
- Track altitude bands separately:
  - 5-15 m
  - 15-30 m
  - 30-60 m
  - 60-100 m

Annotation requirements for your custom dataset
- timestamp
- ground truth latitude / longitude
- altitude AGL
- heading
- camera intrinsics if known
- area type
- weather / lighting notes
- reference map source and date

Milestone 1. Data Ingestion and Geo-Crop Simulator

Goal
Build the geometry and preprocessing scaffold that converts packet data plus prior into a local search crop.

AI Agent
- Implement packet reader for:
  - frame
  - altitude
  - heading
  - approximate prior
- Implement camera geometry utilities:
  - image footprint estimate from altitude and camera calibration
  - meters-per-pixel estimates for the local map
  - heading normalization
  - explicit flat-ground approximation
  - explicit handling for the "camera is nadir" assumption
- Implement a first-pass normalization model:
  - if only altitude and heading are available, treat the frame as nadir and use scale plus rotation normalization only
  - if pitch, roll, or intrinsics become available later, allow a perspective warp path behind a config flag
- Implement geometry sensitivity tests:
  - altitude perturbation sweep
  - heading perturbation sweep
  - output how much crop/scale error those perturbations induce
- Implement local map crop generation around the prior:
  - configurable search radius
  - crop padding
  - rotated crop variants or heading-normalized frame
- Implement visual debugging:
  - show prior center
  - show crop window
  - show expected frame footprint
- Implement a simulator mode using known map coordinates and synthetic perturbations.
- Implement a basic packet replay tool early:
  - read stored packets from disk
  - run them through the preprocessing stack
  - emit timestamped debug outputs
- Implement a minimal live receiver stub early:
  - accept one dev-format packet over the network
  - validate parsing, timestamps, and image decode
  - return an acknowledgment plus parsed metadata

Human
- Provide camera specs if known:
  - resolution
  - focal length or FOV
  - mounting assumptions
- Gather or point to one reference map source for a small test area.
- Validate whether altitude is truly AGL or only approximate above takeoff.

Runnable output
- Given one sample frame packet and prior, the system creates a local map crop, normalized frame view, and debug visualization.
- A replay command can read stored packets and exercise the ingestion path.
- A minimal live receiver can accept and parse a single-session dev packet.

Success metrics
- Crop generation works across several altitudes.
- Visual overlays look geometrically reasonable.
- The expected footprint scales correctly with altitude.
- Under simulated metadata error, the plan reports the induced scale or alignment error instead of hiding it.

Exit criteria
- The system can consistently produce local search regions from real metadata.
- Packet replay works before any neural training starts.
- The minimal live packet path has been exercised before neural training starts.


Milestone 2. Classical Baseline Localizer

Goal
Build a non-neural localizer first. This is the benchmark every neural stage must beat.

AI Agent
- Implement baseline matching with one or more of:
  - ORB
  - AKAZE
  - NCC fallback
- Implement geometric verification:
  - RANSAC homography or affine model
  - inlier counting
  - reprojection error
- Convert the geometric alignment into predicted global position.
- Produce confidence heuristics from:
  - inlier count
  - inlier ratio
  - reprojection error
  - NCC peak sharpness if used
- Build evaluation code on a small labeled dataset.
- Generate failure galleries grouped by failure type.

Human
- Provide a small labeled validation set:
  - at least one flight or short sequence with ground truth
  - ideally multiple altitudes
- Review failure galleries and categorize obvious issues:
  - blur
  - bad map date
  - featureless terrain
  - wrong altitude assumptions

Runnable output
- A baseline localizer that predicts position on labeled data and outputs metrics and overlays.

Success metrics
- Non-trivial Recall@5m on structured terrain.
- Known failure cases are visible rather than hidden.
- Confidence values separate many good matches from bad matches.

Exit criteria
- There is a recorded baseline run in `experiment-log.csv` that future models can compare against.


Milestone 3. Pretrained Neural Matcher Benchmark

Goal
Measure the best zero-shot neural baseline before collecting custom training data or committing to a trainable architecture.

AI Agent

  - RoMa first
  - EfficientLoFTR as fallback or comparison
- Evaluate the pretrained matcher on:
  - public datasets if available
  - the earliest local labeled samples if available
- Build visualizations for:
  - matched correspondences
  - refined position estimate
  - failure cases grouped by terrain and altitude
- Record runtime and GPU/CPU behavior.
- Compare zero-shot neural results directly against the classical baseline.

Human
- Run the zero-shot benchmark on the available machine.
- Review whether the matcher is operationally promising before spending time on custom data collection.

Runnable output
- A zero-shot neural benchmark run with metrics, overlays, runtime numbers, and direct comparison to the classical baseline.

Success metrics
- The neural baseline runs end to end without training.
- The team has evidence about whether pretrained dense matching is viable in the actual regime.
- Failure cases are visible by altitude and terrain type.

Exit criteria
- There is a recorded zero-shot neural run that can be compared directly to the classical baseline.
- A go / no-go decision is written to `experiments/change-log.md`:
  - if zero-shot is close to target, proceed with matcher fine-tuning
  - if zero-shot is clearly inadequate, proceed with a crop-conditioned localizer first
  - if neither path is justified yet, stay in data and geometry investigation

Milestone 5. First Trained Neural Model

Goal
Train the first learned model only after the zero-shot matcher and the dataset pipeline are both in place.

When to do this
Only after Milestone 3 and Milestone 4 are complete.

Decision rule
- If the zero-shot matcher is already close to target accuracy, fine-tune that matcher first.
- If the zero-shot matcher is clearly inadequate but the local crop setup is sound, train a crop-conditioned localizer first.

Recommended order
1. fine-tune the best pretrained matcher
2. if the matcher is far from useful, implement a crop-conditioned localizer before attempting matcher fine-tuning
3. if both are useful, keep the localizer as coarse stage and matcher as refinement stage

AI Agent
- Implement the chosen trained path:
  - matcher fine-tuning, or
  - crop-conditioned localizer training
- If fine-tuning the matcher path:
  - derive a 2D position-in-crop target from ground-truth pose, altitude, heading, and the frame normalization model
  - start by training a small localization head on top of frozen matcher outputs
  - only unfreeze matcher layers if the frozen-feature path is promising but insufficient
- For a crop-conditioned localizer, prefer one concrete starting architecture:
  - dual encoder plus correlation or heatmap head
- Build evaluation, checkpointing, and overlay tooling.
- Add uncertainty estimation from match quality features or output confidence.
- Benchmark the trained model against:
  - classical baseline
  - zero-shot neural baseline

Human
- Run training or finetuning if required.
- Review whether the accuracy gain justifies the latency and implementation cost.

Runnable output
- A trained neural run with measurable gains or a documented rejection of the trained path.

Success metrics
- Better Recall@5m and better p90 error than the best earlier baseline in at least one priority regime.
- Acceptable runtime on the ground computer.

Exit criteria
- There is clear evidence this stage is better than the best earlier baseline, or it is explicitly rejected with recorded evidence.


Milestone 6. Temporal Filter and Sequence Localization

Goal
Fuse per-frame estimates over time so localization is stable and can survive individual bad frames.

Recommended first approach
Particle filter.

State
- x
- y
- heading if needed
- optional velocity proxy if available

Measurement input
- predicted position from matcher/localizer
- confidence
- uncertainty proxy

AI Agent
- Implement the particle filter.
- Convert model confidence and residual scores into measurement likelihood.
- Add state propagation using simple motion assumptions.
- Add sequence evaluation tools.
- Visualize particle clouds, track paths, and recovery after mismatch.

Human
- Provide or validate short flight sequences for sequence testing.
- Review track stability, relocalization behavior, and lag.

Runnable output
- A sequence localizer that consumes a stream of packets and outputs a tracked position over time.

Success metrics
- Better track_median_error_m than frame-only mode.
- Lower jitter during hover.
- Improved robustness during temporary mismatches.

Exit criteria
- Sequence mode is measurably better than independent-frame mode on held-out tracks.

7. Immediate Neural-Network Design Recommendation
==================================================

First neural step to implement
Evaluate a pretrained dense matcher on local crops before committing to any custom training.

Why this is now the first neural step
- It produces signal immediately.
- It uses existing pretrained robustness to correspondence and domain gap.
- It can be benchmarked before local data collection is mature.
- It gives the project an honest "best available off-the-shelf" reference point.

Input
- drone frame, resized
- overhead crop centered on the prior, covering maybe 200-300 m with margin
- heading and altitude as auxiliary numeric inputs

Output
- matched correspondences or refined pose estimate inside the crop
- scalar confidence or match quality score

First trained neural model if training is justified
Use a local crop-conditioned predictor or fine-tuned matcher, depending on Milestone 3 results.

Preferred order after the zero-shot benchmark
1. Fine-tune the best pretrained dense matcher if it is close to useful.
2. Train a crop-conditioned localizer if the matcher path is not good enough or not efficient enough.
3. Use a two-stage system only if it improves Recall@5m by at least 0.05 absolute or p90_error_m by at least 10% without violating the latency budget.

Do not start with
- direct latitude / longitude regression from a full region map
- very large global retrieval architecture
- reinforcement learning for navigation
- end-to-end flight control

9. Recommended Acceptance Targets
==================================================

These are initial targets, not promises. They should be fixed before milestone execution, not redefined after results are known.
Once local labeled data exists, local-data gates supersede public-dataset gates for model-selection decisions.

Phase targets
- Milestone 1 geometry and replay:
  - preprocessing and replay complete without crashes on a packet sample set
  - simulated altitude error of +/-10% must visibly change scale as expected and be reported in diagnostics

- Milestone 2 baseline:
  - Recall@10m >= 0.25 on structured validation scenes
  - p90_error_m < 30 m on structured validation scenes
  - better than trivial center guess by a recorded margin

- Milestone 3 zero-shot neural benchmark:
  - either Recall@5m improves by at least 0.10 absolute over Milestone 2
  - or Recall@10m >= 0.50 on structured validation scenes
  - runtime is measured on the intended machine, not estimated

- Milestone 5 first trained neural model:
  - Recall@5m improves by at least 0.05 absolute over the best earlier baseline in one priority regime
  - p90_error_m improves by at least 10%
  - expected_calibration_error <= 0.10 if confidence is used downstream

- Milestone 6 sequence system:
  - track_median_error_m improves by at least 15% over frame-only mode
  - estimate_jitter_m during hover or low-speed motion improves by at least 25%
  - relocalization_time_s is measured and reported

- Milestone 9 readiness target:
  - Recall@5m >= 0.60 in prioritized environments
  - p90_error_m <= 15 m in prioritized environments
  - failure_rate <= 0.15
  - expected_calibration_error <= 0.10 for any model whose confidence feeds the filter
  - known degraded-mode behavior in difficult low-altitude conditions
  - end_to_end_latency_ms recorded on the target machine

10. Main Risks and How the Plan Handles Them
==================================================

Risk 1
Low-altitude weak-texture scenes may be fundamentally hard to localize.

Mitigation
- baseline early
- subgroup metrics by terrain and altitude
- do not hide failures inside aggregate averages

Risk 2
Reference imagery quality or age may dominate performance.

Mitigation
- log map source and date
- evaluate by map age
- prefer orthophotos where available

Risk 3
Ground-truth labels may be noisy.

Mitigation
- build dataset QA tools before serious training
- require `label_audit.py` pass / fail output before training
- require human sign-off in `experiments/change-log.md`
- split by flight and area
- manually inspect samples every dataset version

Risk 4
A large model may improve accuracy but break latency.

Mitigation
- record runtime metrics from the start
- keep simpler baselines active
- only accept added complexity if it wins materially

Risk 5
The team may lose track of why performance changed.

Mitigation
- mandatory experiment log
- mandatory change log
- per-run artifacts
